# AI协作文档体系更新总结

## 更新目标

根据项目需求（200-300+ syscall，多架构支持）和AI协作特点，重构文档体系：
- **抽象化原则**：具体规则抽象为通用模式
- **前瞻性规划**：基于长期目标而非短期实现
- **灵活适应性**：支持测例驱动的优先级调整

## 主要变更

### 1. 新建 `CLAUDE.md`（简洁导航）
- **核心约束**：core/修改限制（最高优先级）
- **快速导航**：指向详细文档，不重复内容
- **架构原则**：精简版，详细内容在各专门文档中
- **工作流程**：验证门、变更流程简化表达

### 2. 更新 `doc/ai/README.md`
- **双工作流**：区分标准流程（linux_layer/）和特殊流程（core/）
- **强调抽象**：抽象原则优先于具体规则
- **项目目标**：明确验证AI快速构建能力的目标

### 3. 重构 `doc/linux_compat/SYSCALLS.md`
- **阶段划分**：从Step N改为Phase 0-7的阶段性规划
- **灵活性指导**：测例驱动的优先级调整
- **状态跟踪**：添加进度跟踪表
- **长期规划**：面向200-300+ syscall的目标

### 4. 抽象化 `doc/ai/AI_CHECKLIST.md`
- **元规则优先**：§0抽象原则保持不变
- **模式抽象**：具体规则→通用模式（如"命名一致性"→"角色优先命名"）
- **关键原则**：每个部分添加"Key principle"总结
- **示例保留**：Pattern Log保留具体历史案例

## 文档架构理念

### 分层原则
```
CLAUDE.md (稳定导航)
├── 核心约束（很少变动）
├── 快速导航（指向专门文档）
└── 工作流程简化

doc/ai/ (协作文档)
├── AI_CHECKLIST.md (抽象模式)
├── INVARIANTS.md (运行时真理)
├── DECISIONS.md (追加式决策记录)
└── ASSIST_HISTORY.md (变更历史)

doc/linux_compat/ (设计文档)
├── SYSCALLS.md (前瞻性规划，易变)
├── ARCHITECTURE.md (架构原则，稳定)
└── MM_AND_COW.md (技术设计，中等稳定)
```

### 抽象层次
1. **元规则**：最抽象，适用于所有场景（如"抽象优于具体"）
2. **模式**：中等抽象，适用于某类场景（如"并发所有权"）
3. **示例**：具体案例，用于说明模式（Pattern Log）

## 更新哲学

### 抽象化
- **从**：`nexus_node* n` 参数命名错误
- **到**：标识符作用域不应重用typename

### 前瞻性
- **从**：按Step N顺序实现syscall

### 灵活性
- **从**：严格的实现顺序
- **到**：推荐顺序+灵活调整指导
