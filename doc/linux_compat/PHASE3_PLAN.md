# Phase 3: COW和进程复制 - 实现计划

## 目标
实现fork syscall和COW机制，支持多进程应用

## 技术挑战
1. **页故障处理** - 需要trap处理基础设施
2. **COW机制** - 写时复制，物理页分裂
3. **进程复制** - vspace创建和复制
4. **SMP安全** - 多核环境下的内存管理

## 渐进式实现策略

### 阶段A：简化fork（无COW）- 基础版本

**目标**：建立fork基本框架，验证进程创建流程

**实现要点**：
- 创建新Tcb_Base和VS_Common
- 直接复制所有物理页（无共享）
- 设置子进程返回值（子返回0，父返回子pid）
- 基本的进程树管理

**优势**：
- 避免复杂的COW逻辑
- 快速验证fork流程
- 为后续COW打基础

**限制**：
- 内存使用翻倍
- fork性能较差
- 不适合大进程

**文件**：
- `linux_layer/proc/sys_fork.c` - fork实现
- `linux_layer/proc/proc_registry.c` - 进程登记簿

**依赖**：
- core/的vspace创建API
- core/的页表复制API
- 进程管理数据结构

### 阶段B：COW基础设施

**目标**：建立COW机制的基础设施

**实现要点**：
1. **页故障处理框架**
   - 在trap路径中注册COW处理函数
   - 识别COW页故障
   - 调用COW处理逻辑

2. **COW标记机制**
   - 在nexus中标记COW页
   - 在页表项中设置只读属性
   - 引用计数管理

3. **物理页分裂**
   - 检测写访问COW页
   - 分配新物理页
   - 复制内容
   - 更新映射

**文件**：
- `linux_layer/mm/cow_fault.c` - COW页故障处理
- `linux_layer/mm/cow_page_split.c` - 物理页分裂
- `core/` 可能需要添加页故障回调接口

**依赖**：
- 页故障处理基础设施
- nexus的COW标记支持
- 物理页管理API

### 阶段C：完整COW fork

**目标**：实现完整的COW fork机制

**实现要点**：
1. **共享只读页表**
   - 父子进程共享页表
   - 设置正确的引用计数
   - COW标记传播

2. **按需分裂**
   - 写时触发页故障
   - 自动分裂物理页
   - 更新映射关系

3. **性能优化**
   - 批量处理页故障
   - 减少不必要的复制
   - SMP安全优化

**文件**：
- 完善阶段A和B的文件
- 添加性能统计和监控

## 当前状态评估

### ✅ 已完成
- mmap/munmap/mprotect - 内存管理基础
- 进程数据模型 - proc_compat结构
- 测试框架 - 用户态测试运行器

### 🔄 需要实现
- [ ] 页故障处理基础设施
- [ ] COW标记和分裂机制
- [ ] fork syscall
- [ ] vspace复制
- [ ] 引用计数管理

### ⚠️ 潜在风险
1. **core/依赖** - 可能需要core/添加页故障回调接口
2. **SMP安全** - 多核环境下的内存管理复杂性
3. **性能** - COW机制的性能影响
4. **正确性** - 边界情况处理

## 建议的下一步

### 选项1：从简化fork开始
**优势**：
- 快速验证fork流程
- 避免初期复杂度
- 建立测试基础

**劣势**：
- 需要后续重构
- 性能问题

### 选项2：直接实现完整COW
**优势**：
- 一次性解决
- 性能最优
- 架构清晰

**劣势**：
- 复杂度高
- 调试困难
- 可能需要core/支持

### 选项3：混合方式
**策略**：
- 先实现页故障处理基础设施
- 在简化fork基础上添加COW标记
- 逐步完善COW机制

**优势**：
- 平衡复杂度和进度
- 渐进式验证
- 灵活调整

## 推荐方案：选项3 - 混合方式

### 第一阶段：基础设施
1. 分析core/的页故障处理能力
2. 设计COW标记机制
3. 实现基本的页故障处理

### 第二阶段：简化fork + COW标记
1. 实现基础fork（直接复制）
2. 添加COW标记支持
3. 建立测试框架

### 第三阶段：完整COW
1. 实现按需页分裂
2. 优化性能
3. 完善边界情况
