# Core/ 修改策略和优先级

## 🎯 新的约束条件

**原约束**: 不能修改core/代码
**新约束**: 可以在一定程度上修改core，需要审查

## 🔍 需要core/支持的功能

### 1. 内存管理API

#### vspace复制功能
**需求**: fork需要复制父进程的地址空间
**当前状态**: core/缺少vspace复制API
**建议修改**:
```c
// 在vmm.h中添加
error_t copy_vspace(VS_Common* src_vs, VS_Common* dst_vs, 
                    struct map_handler* handler);
```

#### 页表复制优化
**需求**: 高效复制页表结构
**建议修改**:
```c
// 在map_handler.h中添加
error_t copy_user_page_table(VS_Common* src_vs, VS_Common* dst_vs,
                             struct map_handler* src_handler,
                             struct map_handler* dst_handler);
```

### 2. ELF加载支持

#### brk信息传递
**需求**: 正确设置进程的brk值
**当前状态**: linux_elf_init_handler可能未被正确调用
**建议修改**: 
- 确保ELF加载器正确调用init handler
- 或者在gen_task_from_elf中显式传递brk信息

### 3. 页故障处理

#### COW页故障回调
**需求**: fork后写时复制需要页故障处理
**建议修改**:
```c
// 在trap处理中添加页故障回调注册
typedef error_t (*page_fault_handler_t)(vaddr fault_addr, u64 error_code);

void register_page_fault_handler(page_fault_handler_t handler);
```

## 📋 修改优先级

### 高优先级 (立即需要)
1. **vspace复制API** - fork的基础
2. **brk初始化修复** - 堆管理的基础

### 中优先级 (Phase 3需要)
1. **页故障处理回调** - COW的基础
2. **页表复制优化** - fork性能

### 低优先级 (性能优化)
1. **批量内存操作** - 性能提升
2. **缓存优化** - 减少TLB刷新

## 🛠️ 修改流程

1. **需求分析** - 分析Linux兼容层的具体需求
2. **接口设计** - 设计符合core/风格的API
3. **实现方案** - 制定详细的实现计划
4. **用户审查** - 提交方案等待审查
5. **实施** - 审查通过后实施
6. **测试** - 确保修改不影响现有功能

## 🔎 实施建议

### 渐进式策略
1. **先小后大** - 从小的修改开始，建立信任
2. **充分测试** - 每个修改都要有充分的测试
3. **文档同步** - 修改后立即更新文档
4. **向后兼容** - 确保不影响现有功能

### 审查准备
在提交修改前，准备：
1. **需求说明** - 为什么需要这个修改
2. **接口设计** - API的详细设计
3. **实现方案** - 具体的实现计划
4. **测试计划** - 如何验证修改正确性
5. **风险评估** - 可能的影响和缓解措施

## 📊 预期影响

### 正面影响
1. **功能完整性** - 可以实现完整的Linux语义
2. **性能提升** - 在core/层面优化更高效
3. **架构清晰** - 接口在合适的层次实现

### 风险控制
1. **接口稳定** - 确保新接口有良好的扩展性
2. **向后兼容** - 不破坏现有代码
3. **测试覆盖** - 充分的测试确保稳定性
