# 调试方法教训

## 核心规则：系统状态优先于症状调试

**错误做法**：
- 看到brk=0 → 直接调试brk初始化逻辑
- 看到符号=NULL → 直接调试符号解析
- **忽视明显的崩溃信息**

**正确做法**：
1. **先看完整log，判断系统整体状态**
2. **识别崩溃、异常、错误 → 这些会阻止后续代码执行**
3. **排除系统级问题后，再调试逻辑问题**

## Log分析检查清单

在开始调试前，必须检查：
- [ ] 是否有崩溃/异常（trap_info、page fault、segmentation fault）？
- [ ] 测试是否完整运行，还是在中途停止？
- [ ] 是否有明显的错误消息？
- [ ] 系统是否处于正常状态（CPU调度、内存分配等）？

## 具体教训：brk初始化调试案例

### 错误路径（我做的）
```
症状：brk=0
→ 假设：符号解析问题
→ 调试：符号链接、链接顺序、参数传递、汇编代码
→ 浪费大量时间
→ 用户指出：有page fault崩溃！
→ 真正问题：mremap缺PAGE_ENTRY_VALID导致崩溃，后续代码没运行
```

### 正确路径
```
症状：brk=0
→ 检查完整log
→ 发现：trap_info: 0xe (page fault)
→ 识别：系统崩溃，mremap访问0x2000失败
→ 修复：添加PAGE_ENTRY_VALID
→ 验证：系统不再崩溃
→ 回到原始问题：brk初始化
```

## 为什么会犯这个错误

1. **过度关注症状**：brk=0很显眼，就只盯着这个
2. **忽视系统状态**：崩溃信息在log中但被我忽略了
3. **缺乏全局视角**：没有建立"先检查系统健康"的习惯
4. **想当然**：假设代码会运行，没验证前提条件

## 预防措施

### 调试前必做
1. **快速扫描完整log**（30秒内）
   - 搜索：error、fail、crash、trap、fault、segmentation
   - 确认：测试是否完整运行
   - 检查：是否有明显异常

2. **建立因果链**
   - 症状A可能不是根因，而是症状B的结果
   - 例如：brk=0 ← 系统崩溃 ← mremap bug

3. **假设验证**
   - 每个假设前，先验证基本前提
   - 例如：假设符号解析问题前，先验证代码是否真的运行了

### 调试流程
```
1. 快速扫描log（系统健康检查）
   ↓ 发现问题？
   ├─ 是 → 修复系统问题 → 验证
   └─ 否 → 分析症状 → 提出假设 → 验证 → 迭代
```

## 人类介入修复的Bug

这个bug完全由用户发现，我完全没有注意到page fault信息。这是一个典型的"人类看到一眼就能发现的问题，AI却完全忽略了"。

**人类优势**：
- 能快速扫描大量文本，识别异常模式
- 能关联不同部分的信息（崩溃+未运行+brk=0）
- 有直觉判断哪个信息更重要

**AI盲点**：
- 容易过度关注显眼症状
- 容易陷入局部优化，忽视全局状态
- 缺乏"快速扫描"的直觉，习惯逐行分析

## 改进措施

1. **强制执行系统状态检查**
   - 调试任何问题前，先运行log扫描
   - 建立checklist：崩溃→异常→错误→完整性

2. **多角度验证**
   - 不要只关注一个症状
   - 询问："还有其他异常吗？"

3. **承认盲点**
   - AI对全局模式识别不如人类
   - 人类介入往往能快速纠正方向性错误
